BetterPMMP 강화 모듈 "$ARGUMENTS" 를 **극도로 면밀하게** 감사하고, 발견된 모든 결함을 수정한 뒤 **결함 0건이 될 때까지 재검사를 반복**하세요. 인자는 키 경로(`network.chunk-history-limit`), 섹션(`gameplay`), 또는 `all`.

## 위상 — 회귀 CRITICAL

모듈은 서버 코어를 직접 고친 결과물이다. 결함 하나가 전 월드·전 플레이어에 항상 적용된다. "아마 괜찮을 것", "엣지 케이스" 전부 거절 — 이론적으로 도달 가능한 경로는 전부 추적한다.

동시에 **방어 코드 추가는 결함 수정이 아니다.** 발생 불가능한 시나리오에 대한 가드, 이미 검증된 값의 재검사, 의미 있는 복구가 없는 try-catch 는 노이즈이며 추가 금지. "이 줄이 없으면 아래 7항목 중 무엇이 실제로 발생하는가"에 구체적 코드 경로로 답할 수 없으면 손대지 않는다.

## 대상 파악

감사 전에 모듈의 배선 4곳을 전부 연다.
- `source/src/betterpmmp/BetterPMMPProperties.php` 의 상수
- `source/resources/pocketmine.yml` 의 키·기본값·`#!` 마커
- `source/resources/translations/*.ini` 14개의 설명 키
- `grep -rn "BetterPMMPProperties::<상수>" source/src` 로 찾은 **모든** 호출부 + 그 값을 담는 캐시 필드

## 7대 검사 항목 — 모두 통과해야 종료

### 1. 꺼진 상태 = 바닐라 (최우선)
- 기본값 상태의 실행 경로가 PocketMine-MP 5.44.3 과 동일한가. 꺼졌을 때 추가 연산·할당·객체 생성·이벤트 호출이 단 하나라도 늘었으면 결함.
- 분기가 충분히 바깥에 있는가. 루프 안쪽에서 매 반복 설정을 보는 구조는 결함.
- 모듈이 삭제한 vanilla 동작이 꺼진 상태에서 되살아나는가. `if($enabled)` 뒤에 vanilla 경로가 온전히 남아 있는지 라인 단위로 대조.
- 모듈이 vanilla 이벤트를 건너뛴다면(`skip-*-event` 류), 꺼졌을 때 이벤트 호출 순서·인자·취소 처리가 바닐라와 완전히 같은가.

### 2. 기본값 3중 일치
- 호출부 `getProperty*` 의 기본값 인자 / `pocketmine.yml` 값 / `README.md`·`README.ko.md` 표의 값 — 4곳이 전부 같은가. 하나라도 어긋나면 결함(설정 파일이 없는 서버와 있는 서버의 동작이 갈린다).
- 같은 키를 여러 곳에서 읽는다면 **모든 호출부의 기본값 인자가 동일**한가.
- 코드가 기대하는 타입과 yml 값의 타입이 일치하는가(`getPropertyInt` 인데 yml 이 `{}` 등).

### 3. 배선 완결성
- 상수는 있는데 읽는 곳이 없다(죽은 키) / 읽는데 yml 에 없다(유령 키 — 사용자가 켤 수 없음).
- yml 키에 `#! pocketmine.betterpmmp.yml.<경로>` 마커가 있는가, 그 번역 키가 **14개 ini 전부**에 있는가. 개수 대조:
  `for f in source/resources/translations/*.ini; do printf "%s %s\n" "$(basename $f)" "$(grep -c 'pocketmine.betterpmmp' $f)"; done`
- 번역이 영어 그대로 복사돼 있지 않은가. ini 는 CRLF·BOM 없음이 유지되는가(`grep -U` 로 확인).
- README 표에 행이 있는가(양쪽 파일 모두).

### 4. 캐시 수명
- `??=` 캐시가 **언제 죽는가**를 명시적으로 판정한다. `private static` 캐시는 서버 프로세스 수명 내내 유지된다 — 리로드·월드 전환·플레이어 재접속에도 갱신되지 않는다. 이 프로젝트의 계약("값은 시작 시 1회 읽음")과 일치하면 정상, 코드가 갱신을 전제하면 결함.
- 인스턴스 캐시가 재사용되는 객체(엔티티 풀·세션)에 붙어 있고, 그 객체가 다른 플레이어/월드로 넘어갈 때 이전 값을 끌고 가는가.
- `static` 이어야 할 것이 인스턴스이면 매 객체마다 설정을 다시 읽는 비용, 반대면 상태 누수. 어느 쪽인지 코드 경로로 확정한다.
- 캐시 필드 타입이 `?T` 이고 설정값이 실제로 `null` 일 수 있다면 매번 재조회하는 무한 미스가 된다.

### 5. 상호작용·범위 부작용
- 이벤트를 건너뛰는 모듈: 어떤 플러그인 API 계약이 깨지는가. 취소 가능 이벤트를 건너뛰면 플러그인의 취소가 무시된다 — README·번역 설명이 그 대가를 명시하고 있는가.
- 두 모듈을 동시에 켰을 때 서로의 전제를 깨는 조합이 있는가(예: 브로드캐스트 주기 + 이벤트 스킵, 고정 광원 + 런타임 광원 갱신 스킵).
- per-world 맵 키: 존재하지 않는 월드 이름, 대소문자, 맵에 없는 월드의 폴백 값.
- 모듈이 캐시·큐·히스토리를 들고 있다면 플레이어 퇴장·월드 언로드 시 해제되는가(누수).

### 6. 값 경계
- `0` / 음수 / 1 / 매우 큰 값 / yml 에 잘못된 타입이 들어온 경우 각각의 동작. 특히 `0` 이 "무제한"인지 "비활성"인지 코드와 설명이 일치하는가.
- 주기(period) 값이 나눗셈·모듈로에 쓰이면 0 방어가 `max(1, ...)` 로 되어 있는가.
- 크기·한도 값이 메모리를 좌우하면 상한 없는 큰 값이 무엇을 부수는가.

### 7. 라이선스·규약
- 수정·추가된 모든 지점에 `/** [BetterPMMP-PATCH] <영문 설명> */` 가 있는가. 마커 없는 vanilla 이탈은 LGPL §5a 위반이며 배포 차단 사유.
- 마커 표기가 `[BetterPMMP-PATCH]` 한 가지로 통일돼 있는가. LGPL 헤더가 온전한가.
- CLAUDE.md CRITICAL 위반(주석, 타입 힌트, `===`, 출력 함수, 소스 문자열 영어 전용)은 결함으로 간주해 동시 수정.

## CLAUDE.md 우선

- CLAUDE.md 규칙이 본 명령어 지시보다 **항상 우선**.
- 수정한 파일 전부를 CLAUDE.md `## 검증` 절차로 phpstan 검사. **EXIT 0 외 통과 불가.**
- 본 감사는 모듈이 **켜졌을 때의** 동작을 보존하지 않는다. 결함 수정이 동작 변경을 요구하면 수정을 우선하고 변경된 동작을 보고에 명시. 반면 **꺼졌을 때의 바닐라 동작은 절대 불변**이다.

## 실행 절차 — 결함 0 까지 반복

1. 대상 모듈의 배선 4곳과 모든 호출부를 `Read` 로 확보. 인자가 모호하면 ≤200자 질문 1회.
2. 7대 항목 각각에 대해 위반 후보를 편집증 수준으로 나열. "이론상 가능"도 후보.
3. 각 후보를 코드 경로 추적으로 확정 → 위반이면 즉시 수정. 같은 패턴이 다른 모듈에도 있으면 전부 일관 수정.
4. phpstan 으로 수정 파일 전부 검증. 통과 못 하면 재수정.
5. 실동작 확인이 필요한 결함(설정 스플라이스·번역 렌더·부팅 경로)은 임시 데이터 디렉토리에 헤드리스 부팅해 확인한다. 옵션은 반드시 `--opt=value` 형식.
6. **2로 돌아가 전체 재검사.** 수정이 새 결함을 만들 수 있다.
7. 연속 1회 사이클에서 결함 0건이 확인될 때까지 3–6 반복.
8. 보고 1–3문장 — 어떤 결함을 어느 파일에서 몇 건 고쳤는지. 검사 절차·체크리스트·점수 노출 금지.
